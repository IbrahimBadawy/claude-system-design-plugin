Pick Your Complexity Level

Every project asks up-front: Simple, Medium, or Complex?

The plugin scales ceremony to scope. Small tools get out of your way. Production systems get the full treatment.

⚡
Simple
Quick MVP, small tool, prototype
  • Front + Back only
  • 1 architecture diagram
  • No formal plans
  • Skip 3-gate workflow
  • Code fast
⚙
Medium
Full-stack app, moderate scale
  • Front + Back + Auth + DB
  • 5 core plans
  • Unit + integration tests
  • Basic CI/CD
  • Gates optional
🌟
Complex
Production SaaS, distributed
  • All 10 plans + Master + Roadmap
  • Full 3-gate workflow
  • Infra + monitoring + security
  • Full test pyramid + chaos
  • Multi-tenancy + runbooks

Core + Modules — plug-and-play

Same codebase can run standalone, as a core for other apps, or as a module inside another core. Both directions work.

Direction A: HR as core

hr-system/         CORE
├── core/          identity, auth, orgs, events
├── modules/
│   ├── attendance/   plugged in
│   ├── payroll/      plugged in
│   └── portal/       plugged in
└── app-shell/

HR provides identity + auth. Modules consume via ports.

Direction B: Attendance was standalone first

attendance-system/  standalone
├── core/           (minimal, self-contained)
└── app-shell/

# Later → HR wraps it
hr-system/          CORE
├── core/           (richer identity)
├── modules/
│   └── attendance/  with ACL
└── app-shell/

ACL translates HR's Employee ↔ Attendance's User.

Your Programming Skill Level

Asked right after complexity. Controls how many technical questions Claude asks vs. how many it answers itself.

Low skill → Claude picks everything. High skill → full technical dialogue.

👨‍💼
Non-Programmer
Product owner, founder, designer
  • Plain English, no jargon
  • Only product-level questions
  • Claude picks everything
  • Defaults explained briefly
🌱
Beginner
Junior / learning developer
  • Technical but with explanations
  • 1-2 high-level preferences
  • Claude picks most tech
  • Stack listed at end for review
💻
Intermediate
2-5 yrs experience
  • You pick frontend, backend, DB
  • You pick deploy OS (Linux/Windows)
  • You pick UI style & library (PrimeVue, shadcn, ...)
  • Claude shows options w/ pros/cons
🏆
Professional
Senior / staff engineer
  • Full technical dialogue
  • ORM, caching, queue choices
  • Trade-off tables + ADRs
  • Expects you to push back

Milestone Validation Loop

During /implement, after every milestone Claude runs a 6-step validation before moving on.

1
Automated checks
lint · typecheck · unit + integration tests · build · migrations · security audit
2
Acceptance criteria
Compare against IMPLEMENTATION-ROADMAP.md for this milestone
3
Install missing pieces
Packages, env vars, migrations, generated docs
4
Ask for missing info
Secrets, credentials, design decisions that surfaced mid-build
5
Update PROJECT.md
Milestone table: status + validation date + notes
6
Summarize & wait
Plain English for Beginners, technical for Pros. Wait for "proceed" before next milestone.
Simple
2-3 milestones
Medium
4-6 milestones
Complex
7-10 milestones

3-Gate Workflow

You control every phase transition. Nothing happens until you say so.

Phase 1: Discovery Free discussion — no plans, no code
1
/project add university-system Create project & start discussing the idea
2
Discuss, ask questions, explore options, research open source
3
Diagrams (Mermaid) & research saved to discovery/ folder
Gate 1: You say "start planning"
Phase 2: Planning Structured plans — review & approve
4
/plan create Master plan + 10 detailed sub-plans
5
/plan show → /plan edit → /plan approve all Review, refine, approve
Gate 2: You say "start coding"
Phase 3: Implementation Actual code — following the roadmap
6
/plan implementation Step-by-step coding roadmap
7
Build incrementally, docs generated alongside code
8
/docs generate → /project template Ship with docs, save as reusable template

14 Auto-Invoked Skills

Skills activate automatically based on context. No commands needed.

design-system

Triggers when designing new systems. Uses 4-step framework, references 16 design patterns, searches for open source alternatives.

evaluate-system

Triggers when reviewing systems. Multi-dimensional evaluation using criteria from all 8 books. Score card with prioritized findings.

implement-system

Triggers when coding. Writes real production code. Researches docs before using any library. Follows Clean Architecture.

research-tech

Triggers when comparing technologies. Searches the web, reads official docs, synthesizes findings into comparison matrices.

project-manager

Triggers for project management. Tracks multiple projects, manages plans, templates, and dependencies between projects.

documentation

Triggers when generating docs. Recommends Swagger, Storybook, TypeDoc, Docusaurus based on stack. Auto-detects undocumented code.

frontend-design

Triggers for UI work. Asks about style preferences, picks best component library (shadcn, PrimeVue, MUI, etc.), enforces accessibility.

devops

Triggers for infrastructure. Generates Docker, CI/CD, deployment configs. Supports GitHub Actions, AWS, Kubernetes, Terraform.

knowledge-builder

Triggers for unfamiliar domains. Researches deeply from web + docs, builds structured knowledge files. Import PDFs, manuals, datasheets.

opensource-research

Triggers when looking for existing solutions. Compares open source systems, analyzes licenses (MIT/GPL/AGPL), recommends build vs buy.

web-apps

Triggers for web apps. Framework selection (Next.js, Nuxt, SvelteKit), rendering strategy (CSR/SSR/SSG), PWA, Core Web Vitals, deployment.

desktop-apps

Triggers for desktop apps. Electron, Tauri, WPF, Qt, Flutter Desktop. Packaging, auto-update, code signing, system tray, installers.

mobile-apps

Triggers for mobile apps. React Native, Flutter, SwiftUI, Kotlin Compose. App store submission, push notifications, offline-first.

cross-platform

Triggers for multi-platform. Shared code strategies, monorepo, BFF pattern, platform migration (web→mobile, desktop→web).

30 Always-Active Rules

These rules are enforced automatically on every interaction. Non-negotiable quality standards.

01Requirements First — never code without understanding the problem
02Design Before Code — API + data model + architecture first
03Justify Decisions — every choice needs a "why" with trade-offs
04Think About Failure — timeouts, retries, circuit breakers, degradation
05Security by Default — auth, encryption, validation in every layer
06Clean Code — naming, functions, error handling, DRY, SOLID
07Scalability — design for 10x, build for 1x, don't over-engineer
08Observability — metrics (RED), logging (JSON), tracing, alerting
09Data Model is King — access patterns first, schema second
10API Standards — versioning, pagination, idempotency, validation
11Research Before Code — fetch latest docs before using any library
12Auto Documentation — docs generated with code, auto-update triggers
13No Code Without Plan — all plans approved + explicit user command (Complex)
14Domain Modeling — DDD: bounded contexts, aggregates, events
15Accessibility — WCAG 2.2 AA, keyboard nav, screen readers, RTL
16Performance Budgets — API p99, Core Web Vitals, bundle size limits
17Tenant Isolation — every query tenant-scoped, no cross-tenant data leakage
18Testing Required — test pyramid enforced, coverage targets, no merge without tests
19Suggest Next Commands — after every command, suggest 2-4 relevant next steps
20Chat Visibility — diagrams & code rendered inline, never hidden in files
21Complexity-Aware — scale effort to project complexity, ask if unsure
22Skill-Level-Aware — adapt question depth to user's programming skill
23Milestone Validation — after every milestone: validate, install, ask, then wait
24UX Completeness — every page has 5 states + feedback + keyboard + WCAG 2.2 AA
25RBAC by Default — any app with users gets full roles × permissions + audit log
26Design System First — tokens before UI code, no hex in components, atomic design
27Copy Quality — no lorem ipsum, no "Click here", verb+object buttons, inclusive language
28Data Viz Honesty — bar charts start at 0, no chartjunk, chart type matches data shape
29Modular by Default — every system is Core + Modules, plug-and-play, no cross-module imports NEW
30Functional Completeness — classify capabilities + auto admin UIs + whole app manageable NEW
