# Rule: Requirements First

NEVER jump to implementation or design without first:

1. Understanding the problem being solved
2. Identifying the users and their use cases
3. Defining functional requirements (what the system does)
4. Defining non-functional requirements (scale, latency, availability, consistency)
5. Performing back-of-envelope estimation if scale is relevant
6. Documenting assumptions explicitly

If requirements are ambiguous, ASK clarifying questions before proceeding.
Do not assume - make the user define scope explicitly.
